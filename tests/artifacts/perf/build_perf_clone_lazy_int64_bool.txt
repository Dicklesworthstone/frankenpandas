Lever: Column::clone preserves typed all-valid Int64/Bool lazy backing
File: crates/fp-columnar/src/lib.rs  (clone_dense_values_from_cache)

ROOT CAUSE
  Column::clone() -> clone_dense_values_from_cache():
    - Float64 arm built ScalarValues::lazy_all_valid_float64(data) -> as_f64_slice OK on clone.
    - Int64  arm built ScalarValues::from_vec(map(Scalar::Int64)) -> EAGER Vec<Scalar>:
        * full per-row Scalar materialization on every clone, AND
        * as_i64_slice() returns None on the clone (it only reads ColumnData::Int64
          or ScalarValues::LazyAllValidInt64).
    - Bool arm: same eager pattern; as_bool_slice() lost on clone.
  => Cloned Int64 key/value columns silently MISS every dense / direct-address
     fast path (groupby build_groups, value_counts, factorize, dedup, joins).
     The vs_pandas bench clones the frame, so groupby ran the slow ScalarKey
     hashing build_groups instead of the dense direct-address path.

FIX
  Int64 arm -> ScalarValues::lazy_all_valid_int64(data.clone())   (mirror Float64)
  Bool  arm -> ScalarValues::lazy_all_valid_bool(data.clone())
  (Int64Nullable/BoolNullable fall through to self.values.clone(); precedent =
   Float64 arm which is non-nullable only.)

ISOMORPHISM
  LazyAllValidInt64/Bool materialize the Scalar view via the identical
  data.iter().copied().map(Scalar::Int64|Bool) used by the old eager clone,
  so the observable Scalar sequence is bit-identical; only laziness +
  contiguous-slice availability change. Validated:
    - fp-columnar lib: 377 passed / 0 failed
    - fp-frame    lib: 2823 passed / 0 failed (incl. *_golden_basic bit goldens:
                       value_counts_golden_basic, unique_golden_basic, var_golden_basic ...)
